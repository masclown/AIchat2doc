# 项目开发说明文档

## 目的
修复在使用 Markdown 导出功能时，如果无序列表下方直接紧跟代码块且代码块语言类型名称中包含特殊字符（例如点号 `.`，如 `docker-compose.yml`），会导致缩进去除失效及代码块格式错乱的问题。

## 当前能够实现
- 提取特定支持的多平台问答界面文本，并生成对应的 Markdown 内容。
- 处理简单的内嵌或紧跟在无序列表外的代码块（常规非特殊字符语言标识）的渲染。
- 去除 Markdown 代码块缩进以恢复原有代码缩进显示效果。

## 具体实现
1. 定位到 `content.js` 文件中对于清理和重新排版受列表影响的代码段逻辑：
    - 此前的正则表达式：`/([^\n]*?)[ \t]*```([a-zA-Z0-9_+#-]*)\n([\s\S]*?)\n([ \t]*)```/g` 仅匹配字母、数字、下划线、加号、井号和连字符的语言类别名。
    - 当遇到 `docker-compose.yml` 这类带 `.`（点号）的文件扩展名式语言声明时，会导致整个替换清理过程失效，从而原样保留被自动生成的极长缩进前缀空格。
2. 更改提取规则：
    - 更新后的正则表达式：`/([^\n]*?)[ \t]*```([^\s]*)\n([\s\S]*?)\n([ \t]*)```/g`
    - 改为匹配任何非空字符序列作为 `language` 部分，更稳健地适应各种被标记的代码段标头声明。

## 已解决问题
- 修复了用户在遇到类似于 `docker-compose.yml` 等带标点符号语言标注的代码段出现在列表层级内时，无法被工具正确处理列表附带过量空格前缀及连字符换行错误的问题。这恢复了其作为普通顶层（0缩进）标准 Markdown 代码块的展示形态。

## AI 讨论记录
- **主要探讨点：** 为什么特定情况的代码块没有执行既定的对齐去除策略？
- **发现和总结：** 我们追踪到了匹配代码标识的正则表达式对于带有 `.` 标志的命名方式没有做兼容。而 Turndown 给 `ul>li>pre` 中代码添加的高额多级列表缩进会由于正则失败逃过消除环节。最终通过放开正则针对换行以外非空白字符的支持进行了一次完美的结构降级恢复。

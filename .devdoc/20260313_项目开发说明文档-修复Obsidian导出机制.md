# 20260313_项目开发说明文档-修复Obsidian导出机制

## 目的
修复两个与 Obsidian 导出相关的问题：
1. **Minimax 平台无法打开 Obsidian**：点击"Save to Obsidian"按钮后 Obsidian 没有响应，库中无导出文档。
2. **长内容导出格式丢失**：当对话内容较长时，导出到 Obsidian 的文档中所有换行符丢失，代码块被压缩成单行，段落间也无分隔。

## 当前能够实现
- 在所有平台上可靠地打开 Obsidian 协议链接（包括 Minimax 等 SPA 站点）。
- 当内容过长（编码后 URL 超过 30000 字符）时，自动将内容复制到剪贴板并提示用户粘贴，避免格式损坏。

## 具体实现
- 重写 `content.js` 中的 `saveToObsidian` 函数：
  1. **替换导航方式**：将 `window.location.href = url` 改为创建隐藏的 `<a>` 标签并触发点击。这避免了 SPA 框架（如 Minimax）的路由拦截导致协议链接无法被系统处理。
  2. **URL 长度保护**：在发送前检查完整 URL 长度，当超过 30000 字符时：
     - 使用 `navigator.clipboard.writeText()` 将完整 Markdown 内容复制到剪贴板。
     - 创建一个仅含占位提示文字的简短 Obsidian 链接。
     - 弹窗提示用户在 Obsidian 中粘贴替换。
  3. **兜底处理**：如果剪贴板 API 不可用，仍然尝试使用完整 URL。

## 当前无法实现/未解决问题
- 超长内容需要用户手动粘贴，无法全自动导入 Obsidian。

## AI 讨论记录
1. **Minimax Obsidian 打不开的原因分析**：Minimax (`agent.minimaxi.com`) 是一个 SPA（单页应用），其前端路由框架会拦截 `window.location.href` 的赋值操作。当目标是 `obsidian://` 这种自定义协议时，路由器无法处理，导致请求被吞掉。解决方法是创建一个不在路由管辖范围内的 `<a>` 元素并模拟点击，浏览器会直接调用操作系统的协议处理器。
2. **长内容格式丢失的原因分析**：使用 `encodeURIComponent` 编码的 Markdown 内容通过 `obsidian://` URI 传递时，操作系统协议处理器对 URL 长度有实际限制（Windows 上约 32767 字符）。超长 URL 会导致内容被截断或换行符 `%0A` 被丢弃，表现为所有 `\n` 消失。解决方案是检测 URL 长度并改用剪贴板作为内容传递通道。

# 项目开发说明文档 - 适配多平台及DeepSeek

## 项目当前状态

**当前能够实现：**
- 快速将 Gemini 和 DeepSeek 的对话页面转换为 Markdown 格式。
- 点击按钮跳转到 Obsidian 并新建一条带时间戳的笔记。
- 点击按钮下载 md 文件到本地。
- 基于平台适配器模式提取并清理不同平台特定 DOM 结构（如表格、代码块等）。
- 完美适配 DeepSeek 的原生代码块结构、多余换行优化以及来源引用链接。

**当前无法实现 / 未解决问题：**
- 需要更多的平台测试以确保完全兼容。

**项目结构：**
- `manifest.json`: 浏览器插件配制文件。增加了 DeepSeek 域名。
- `content.js`: 核心逻辑。已重构为 `PLATFORMS` 适配器模式，包含 `gemini` 和 `deepseek` 配置。
- `styles.css`: 样式文件。
- `turndown.js` / `turndown-plugin-gfm.js`: Markdown 处理库。
- `devdoc/`: 存放项目开发说明文档。

## AI 讨论记录
- **问题1**：用户希望在现有插件基础上适配多个平台的对话导出功能，首个目标为 `chat.deepseek.com`。
- **解决方案1**：重构 `content.js`，抽离出平台专有的适配器 (`PLATFORMS`)，针对 `chat.deepseek.com` 编写页面匹配、按钮注入与内容提取清洗规则。修改 `manifest.json` 授权该网址。
- **问题2（深度优化）**：测试中发现 DeepSeek 代码块无 \`\`\` 符号、导出内容存在过多换行，且正文引用链接渲染为 `-数字` 等影响阅读体验的问题。
- **解决方案2**：
  1. 重新设计 `deepseek.cleanDOM` 中的 `<pre>` 的降级处理逻辑。提取出语言后，清除多余的兄弟节点并显式为其内部构建标准的 `<code class="language-xxx">` 元素，使得 Turndown 能正确识别并输出栅栏代码块。
  2. 对于 DeepSeek 中 `<li>` 随附的 `<p>` 元素转为 `<span>`，并在外围合并多余换行的阶段清空了末尾回车与空格产生的排版干扰空行，大幅压缩多余空行。
  3. 通过正则表达式筛查所有 `<a>` 标签，寻找到纯数字的来源引用形式（包括横线匹配），将其文字直接替换为带有 `🔗` emoji的形式并包裹至 `<sup>`，极大地增强了在 Obsidian 中的排版体验。
- **问题3（格式化问题修复）**：和先前 Gemini 聊天页面出现过相同问题：代码块的语言没有紧跟在 ``` 之后，且代码块末尾的 ``` 后面缺少换行导致排版错乱；另外导出的 MD 标题本身被不必要地用 ** 框住。
- **解决方案3**：
  1. 通过 `turndownService.addRule('fencedCodeBlock', ...)` 显式编写自定义规则，精确提取代码块的 `language-xxx`，并确保输出格式为 `\n\n```language\n代码内容\n```\n\n`，解决前置与后置换行以及代码语言未声明的问题。
  2. 修复 `getMarkdownContent` 中的正则表达式清理逻辑，删除了原先破坏段落和代码块边界的多重空行消除正则，将其统一限制为最大连续两个换行 `\n{3,} -> \n\n`。
  3. 通过正则表达式过滤标题，移除原先 `### **标题内容**` 这类情况中不必要的 `**` 加粗符号。
- **问题4（代码块语言丢失及首行夹杂数字）**：DeepSeek 导出的个别代码块不仅没有正确的语言名称，反而还在代码块 ```` ` 第一行显示了多余的数字，并且外部文字却散落了 `python`。
- **解决方案4**：
  1. 深入分析后发现，原有的代码块 `cleanDOM` 头节点寻址逻辑：`current = pre.parentElement` 后无限向上查找 `previousElementSibling` 存在致命越界缺陷，一旦当前代码块容器内没有直接的兄弟前置节点，就会“跳出”当前区域匹配到上级甚至是上一段普通数字文本 `<p>1</p>` 作为所谓的 “header” 并提取为语言名 `language-1`，同时导致真正的语言标识遗漏为正常文本并跳出代码块渲染。
  2. 重新改写 `cleanDOM`：检索起始点更改为 `current = pre` 本身，限制最大向上追溯3级，并新增校验逻辑，只有前置兄弟节点为 `DIV` 或 `SPAN` 或 `HEADER`，并且文字简短及非纯数字等才确信这是一个代码块头部，提取大写语言名之后执行 `.remove()`，完美解决了越界解析数字和语言标识提取错位的问题。
- **问题5（代码块在列表中造成的粘连与异常内部缩进）**：Gemini导出的内容中，当代码块紧跟在列表及其内容（例如 `<li>` 随附的 `<p>`）之后时会被 Turndown 当作列表子级渲染，导致代码块首行的 ```` ` 前置空白被强行清空拼接到前文最后一行，且整个内部按列表层级被附加了缩进，导致渲染大段溢出错误。
- **解决方案5**：在 `getMarkdownContent` 末端转换生成字符串后增加一层全局正则表达式处理 (`/([^\n]*?)[ \t]*```([a-zA-Z0-9_+#-]*)\n([\s\S]*?)\n([ \t]*)```/g`)：当识别到前面沾粘文字时，截断并在中间插入两个换行符将其强行推入新段落；同时提取末尾反引号行的空白数作为参考缩进量，并去除内部的跟随缩进效果使得代码恢复左对齐原样。
